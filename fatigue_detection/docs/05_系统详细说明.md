# 系统详细说明

## 1. 系统整体架构

系统分成两条主线：

- 在线主线：实时输入 -> 关键点 -> 特征 -> 窗口 -> ONNX 推理 -> 状态平滑 -> 告警 -> UI
- 离线主线：原始视频 -> 逐帧特征 CSV -> 窗口标签 -> 窗口数据集 -> 训练 -> 导出 ONNX -> 实验评估

在线和离线共用同一套特征定义，这一点很重要。它保证了训练阶段和部署阶段的输入含义一致。

## 2. 在线运行逻辑

主入口是 `src/main.py`。运行 `python src/main.py` 时，程序会按下面顺序执行：

1. 从 `src/config.py` 读取默认配置。
2. 创建 `CameraStream`，打开摄像头或视频文件。
3. 创建 `FaceLandmarker`，加载 `face_landmarker.task`。
4. 创建 `FeatureExtractor`，把关键点转换为每帧疲劳特征。
5. 创建 `FeatureWindowBuffer`，维护滑动窗口。
6. 创建 `OnnxFatigueInferencer`，加载 `mlp.onnx` 和 `scaler.pkl`。
7. 创建 `AlertManager`，管理连续告警和冷却逻辑。
8. 创建 `UIOverlay`，显示实时界面。
9. 进入主循环，对每一帧重复执行检测、特征提取、窗口构建、推理和显示。

### 2.1 单帧处理逻辑

每次读取一帧后：

1. 用 MediaPipe 检测人脸关键点。
2. 如果没有检测到人脸，清空窗口缓存和投票缓存，并显示 `No Face`。
3. 如果检测到人脸，计算该帧的 `EAR`、`MAR`、`pitch/yaw/roll`。
4. 把该帧特征送入窗口缓存。
5. 当窗口长度达到 `window_size` 后，构建 15 维窗口特征。
6. 用 ONNX 模型预测当前窗口是否为疲劳窗口。
7. 对最近多个窗口做投票平滑。
8. 满足告警门槛后触发一次提醒。
9. 将状态、置信度和其他指标绘制到画面。

## 3. 特征逻辑

### 3.1 每帧特征

定义在 `src/features.py`：

- `ear_left`
- `ear_right`
- `ear_avg`
- `mar`
- `pitch`
- `yaw`
- `roll`

这些特征都是从人脸关键点直接计算出来的。

### 3.2 窗口特征

定义在 `src/window_buffer.py`，总共 15 维：

1. `ear_left`
2. `ear_right`
3. `ear_avg`
4. `mar`
5. `pitch`
6. `yaw`
7. `roll`
8. `ear_mean`
9. `ear_min`
10. `mar_mean`
11. `mar_max`
12. `low_ear_streak`
13. `high_mar_streak`
14. `close_eye_ratio`
15. `yawn_ratio`

其中：

- `low_ear_streak` 表示窗口尾部连续低 EAR 的长度
- `high_mar_streak` 表示窗口尾部连续高 MAR 的长度
- `close_eye_ratio` 表示窗口中闭眼帧占比
- `yawn_ratio` 表示窗口中高 MAR 帧占比

这些特征比单帧更能表达“持续疲劳”。

## 4. 模型与触发逻辑

当前触发逻辑分成 3 层，不是“单次判断为疲劳就报警”。

### 4.1 第一层：窗口级模型判断

定义在 `src/infer_onnx.py`。

ONNX 模型输出两个类别的概率，程序重点关注 `fatigue_probability`。当前默认规则是：

- `fatigue_probability >= fatigue_confidence_threshold`
- 才把当前窗口记为 `Fatigue Risk`

默认阈值：

- `fatigue_confidence_threshold = 0.78`

### 4.2 第二层：投票平滑

定义在 `src/main.py` 的 `update_vote_state()`。

程序不会直接使用当前窗口的一次结果，而是维护最近 `vote_window_size` 个窗口的判断结果，再统计其中疲劳窗口的比例。

默认参数：

- `vote_window_size = 7`
- `vote_positive_threshold = 0.67`

这意味着最近 7 个窗口里大约至少 5 个为疲劳，最终平滑状态才会变成 `Fatigue Risk`。

### 4.3 第三层：连续告警门控

定义在 `src/alert.py`。

即使平滑状态已经是疲劳，也还要继续满足：

- 当前平滑状态为 `Fatigue Risk`
- `confidence >= alert_confidence_threshold`
- `vote_ratio >= alert_vote_ratio_threshold`
- 连续满足 `consecutive_alert_threshold` 次
- 距离上次告警超过 `alert_cooldown_seconds`

默认参数：

- `alert_confidence_threshold = 0.82`
- `alert_vote_ratio_threshold = 0.75`
- `consecutive_alert_threshold = 3`
- `alert_cooldown_seconds = 3.0`

这几层叠加起来的目的，就是尽量避免正常眨眼引起误报。

## 5. 什么动作更容易触发

更容易触发最终状态或告警的动作：

- 持续闭眼 1 到 2 秒
- 连续明显打哈欠
- 长时间眼睛半闭、嘴巴张开
- 低头且伴随明显眼部闭合或嘴部张开

通常不该触发最终告警的动作：

- 一次正常眨眼
- 一两帧的短暂张嘴
- 短暂低头
- 瞬时表情变化

如果出现误触发，优先调 `config.py` 里的四个值：

- `fatigue_confidence_threshold`
- `vote_window_size`
- `vote_positive_threshold`
- `alert_vote_ratio_threshold`

## 7. UI 逻辑

界面逻辑定义在 `src/ui.py`。

当前支持：

- 默认精简模式
- 详情模式
- 中英文切换
- 中文字体渲染
- 告警横幅

运行时按键：

- `L`：切换中英文
- `I`：显示或隐藏详情
- `Q` / `ESC`：退出

默认界面只显示：

- 状态
- 置信度
- FPS

详情模式增加：

- 原始状态
- 投票比例
- 疲劳概率
- `EAR`
- `MAR`
- `Pose`
- `Backend`
- 状态说明

## 8. 数据与训练逻辑

### 8.1 特征提取

脚本：`train/extract_features.py`

作用：

- 读取原始视频
- 每隔 `frame_stride` 帧取一次样
- 对每个有效帧提取特征
- 输出逐帧 CSV

默认会写出这些字段：

- `frame_index`
- `timestamp_seconds`
- `label`
- `dataset_name`
- `group_id`
- `source_video`
- `backend`
- 每帧特征字段

### 8.2 窗口标签

脚本：`train/generate_window_label_review.py`

作用：

- 对疲劳视频的窗口打一个启发式分数
- 选出更可信的疲劳风险区间
- 输出 `window_labels_review.csv`

这一步不是人工精标，但比整段视频直接继承标签更干净。

### 8.3 建训练集

脚本：`train/build_dataset.py`

作用：

- 读取逐帧特征 CSV
- 每 `window_step` 帧滑动一次
- 生成窗口特征向量
- 根据 `window_labels_review.csv` 决定保留哪些窗口以及它们的标签
- 输出 `dataset_cv6_review.npz`

输出数据包含：

- `X`
- `y`
- `feature_names`
- `sources`
- `groups`
- `label_sources`

### 8.4 训练

脚本：`train/train_mlp.py`

作用：

- 对输入特征做 `StandardScaler`
- 训练轻量 MLP
- 用受试者分组交叉验证评估
- 保存最终全量训练模型

划分逻辑：

- 受试者数较少时用 `LeaveOneGroupOut`
- 受试者更多时用 `GroupKFold`

这保证了同一个人不会同时出现在训练集和验证集。

### 8.5 导出和评估

脚本：

- `train/export_onnx.py`
- `train/evaluate.py`
- `train/rule_baseline.py`
- `train/benchmark_inference.py`
- `train/feature_ablation.py`

它们分别负责：

- ONNX 导出
- 模型评估
- 规则基线对比
- 推理性能测试
- 特征消融实验

## 9. 当前实验结果怎么理解

当前主要结果：

- 分组交叉验证 `accuracy = 0.7238`
- 分组交叉验证 `f1 = 0.5680`
- 规则基线 `f1 = 0.7156`
- ONNX 前向推理 `0.0215 ms/sample`
- ONNX 相对 PyTorch 加速 `3.14x`
- 最优特征组合 `eye_mouth_window`，平均 `f1 = 0.6167`

这些结果说明：

1. 在线部署已经没有性能瓶颈。
2. 当前主要问题不在速度，而在标签质量和泛化能力。
3. 眼部和嘴部统计特征是最稳定的疲劳信号。

## 10. 常见调整建议

如果你要为真实演示继续微调，优先级建议如下：

1. 先调 `fatigue_confidence_threshold`
2. 再调 `vote_window_size` 和 `vote_positive_threshold`
3. 最后调 `alert_vote_ratio_threshold` 和 `consecutive_alert_threshold`
4. 如果画面被遮挡，就保持默认精简界面，不要长期开详情
