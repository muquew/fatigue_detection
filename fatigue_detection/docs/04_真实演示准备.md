# 真实演示准备

## 1. 演示目标

这次真实演示的目标不是证明模型学术上最强，而是稳定展示下面 4 件事：

1. 系统可以本地实时运行
2. 可以从摄像头或视频读取输入
3. 可以输出疲劳相关状态与告警
4. 可以说明训练、部署和实验对比已经完成

## 2. 演示前必须检查的文件

先确认这些文件存在：

- `models/face_landmarker.task`
- `models/mlp.onnx`
- `models/scaler.pkl`

如果缺少 `mlp.onnx`，先执行：

```bash
python train/export_onnx.py
```

## 3. 演示前建议做的 3 次检查

### 3.1 自检

```bash
python src/main.py --dry-run --frames 20
```

目的：

- 检查主程序入口
- 检查 ONNX 是否能加载
- 检查状态平滑和告警是否正常

### 3.2 本地视频检查

```bash
python src/main.py --source 你的测试视频.mp4
```

目的：

- 先在可重复输入上验证显示效果
- 避免一上来就用摄像头排查问题

### 3.3 摄像头检查

```bash
python src/main.py
```

目的：

- 检查摄像头权限
- 检查实时 FPS
- 检查画面叠加和按键退出

## 4. 演示时建议展示哪些动作

建议按这个顺序来：

1. 正常注视镜头
2. 缓慢闭眼 1 到 2 秒
3. 张嘴模拟哈欠
4. 低头或明显姿态变化

这样你更容易解释：

- `EAR` 下降对应闭眼
- `MAR` 升高对应张嘴/哈欠
- 头姿变化会反映在 `pitch/yaw/roll`
- 最终状态由模型输出，再经过投票平滑和连续门控

## 5. 演示时怎么解释界面

当前界面是两级信息：

- 默认精简模式：只显示状态、置信度、FPS 和控制提示，适合录屏
- 详情模式：再显示原始状态、投票比例、疲劳概率、EAR、MAR、头姿和后端

运行时按键：

- `L`：中英文切换
- `I`：精简/详情切换
- `Q` / `ESC`：退出

## 6. 演示时怎么解释“为什么还要训练模型”

建议你用这段话：

“系统不是把原始视频直接丢进大模型，而是先提取眼睛、嘴巴、头姿等疲劳特征，再用一个轻量 MLP 学习这些特征的组合关系。这样既保留可解释性，也更适合边缘设备部署。”

## 7. 演示时怎么解释“为什么规则法还比模型强”

你可以这样解释：

“当前实验子集只有 6 个受试者，虽然已经引入窗口级标签，但标签质量仍然有限。在这种情况下，基于人工经验设计的规则法非常强，这说明当前瓶颈主要在标签质量和跨受试者泛化，不在推理速度。后续如果补更高质量标签，模型会更有优势。”

## 8. 当前最适合你展示的实验结论

你可以重点讲 3 条：

1. ONNX 本地推理速度快，适合边缘部署
2. 窗口级标签比整段视频标签更合理
3. `eye + mouth` 的窗口统计特征比当前全特征输入更稳

## 9. 如果演示时效果不稳定，优先检查什么

按优先级检查：

1. 光照是否太差
2. 脸是否完整出现在画面里
3. 是否只是正常眨眼，而不是持续闭眼
4. 摄像头分辨率和帧率是否异常
5. `face_landmarker.task` 是否正确加载

## 10. 你下一步最值得做什么

如果你现在要准备真实演示，最值得做的是：

1. 用一段本地视频先跑一遍完整展示流程
2. 再用摄像头录一版真实操作
3. 最后决定是否继续调 `config.py` 里的告警阈值
