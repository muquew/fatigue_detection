# 程序设计说明

项目名称：**基于边缘智能计算的轻量化实时疲劳状态检测系统设计与实现**

---

## 1. 项目概述
本项目设计一个运行在本地设备上的轻量化实时疲劳检测系统。系统通过摄像头采集用户面部视频，在本地完成关键点提取、疲劳特征计算、轻量神经网络推理以及告警输出，以实现低时延、低带宽依赖和隐私友好的实时疲劳监测。

项目整体目标如下：

1. 实现摄像头视频流的实时采集；
2. 实现基于人脸关键点的疲劳特征提取；
3. 使用轻量 MLP 模型完成疲劳状态分类；
4. 将模型导出为 ONNX 并使用 ONNX Runtime 进行推理；
5. 设计可视化界面，展示状态、关键指标和告警信息；
6. 完成分类效果与推理性能实验。

## 2. 技术选型

### 2.1 开发语言
**Python**

选择原因：
- 生态成熟，适合课程项目快速实现；
- 便于接入 OpenCV、MediaPipe、PyTorch、ONNX Runtime 等工具；
- 适合完成训练、部署、实验和可视化全流程开发。

### 2.2 视频采集与界面显示
**OpenCV**

主要用途：
- 打开摄像头并读取视频帧；
- 在图像上绘制关键点、状态信息和 FPS；
- 实现程序界面显示与调试。

选择原因：
- 接口简单，适合快速构建实时视觉应用；
- 与 Python 集成方便；
- 能满足课程作业对演示界面的基本需求。

### 2.3 人脸关键点检测
**MediaPipe Face Landmarker**

主要用途：
- 从视频帧中检测人脸；
- 输出眼部、嘴部和面部轮廓等关键点坐标；
- 为后续疲劳特征计算提供输入。

选择原因：
- 实时性较好；
- 适合连续视频流处理；
- 可以快速接入 Python 项目；
- 适合边缘侧轻量视觉任务。

### 2.4 特征处理与标准化
**NumPy + scikit-learn**

主要用途：
- 构建结构化特征向量；
- 使用标准化方法统一输入尺度。

选择原因：
- 疲劳特征本质上是低维数值特征；
- 标准化有助于提高轻量模型训练稳定性；
- 适合小规模结构化数据处理。

### 2.5 分类模型
**轻量 MLP（多层感知机）**

主要用途：
- 学习 EAR、MAR、头姿以及时间窗口统计特征与疲劳状态之间的关系；
- 输出疲劳状态分类结果。

选择原因：
- 模型结构简单；
- 参数量小；
- 推理速度快；
- 适合边缘端部署。

### 2.6 模型训练框架
**PyTorch**

主要用途：
- 定义轻量 MLP 模型；
- 完成训练、验证和权重保存；
- 导出 ONNX 模型。

选择原因：
- 灵活性高；
- 便于自定义模型结构；
- ONNX 导出流程较成熟。

### 2.7 推理部署框架
**ONNX Runtime**

主要用途：
- 加载导出的 ONNX 模型；
- 在本地执行高效推理；
- 作为边缘部署阶段的统一推理接口。

选择原因：
- 部署结构清晰；
- 适合展示“训练—导出—部署”的完整流程；
- 能体现边缘智能中模型轻量部署的特点。

## 3. 技术路线

### 3.1 总体技术路线
系统采用如下处理流程：

**摄像头视频流 → 人脸关键点提取 → 几何特征计算 → 时间窗口特征构造 → 特征标准化 → 轻量 MLP 分类 → ONNX Runtime 推理 → 实时状态显示与告警**

### 3.2 路线说明
本项目不是直接对整幅图像进行端到端分类，而是先通过关键点模型提取人脸几何信息，再将与疲劳相关的低维特征输入轻量分类模型。这种方式有以下优点：

1. 降低输入维度，减轻边缘端计算压力；
2. 提高系统可解释性，便于展示 EAR、MAR 等指标；
3. 更容易完成小规模训练与快速部署；
4. 有利于构建实验对比和分析模型效果。

### 3.3 特征构成
建议输入模型的特征包括以下两部分：

#### 单帧几何特征
- 左眼 EAR
- 右眼 EAR
- 平均 EAR
- 嘴部 MAR
- 头部 pitch
- 头部 yaw
- 头部 roll

#### 时间窗口统计特征
- 最近 N 帧 EAR 平均值
- 最近 N 帧 EAR 最小值
- 最近 N 帧 MAR 平均值
- 最近 N 帧 MAR 最大值
- 连续低 EAR 帧数
- 连续高 MAR 帧数
- 闭眼比例
- 哈欠比例

通过将单帧特征与时间窗口统计特征结合，可以更好地表征疲劳状态的持续性，而不是只依赖某一瞬时帧。

## 4. 系统流程设计

### 4.1 整体流程
系统整体运行流程如下：

1. 初始化摄像头；
2. 加载关键点检测器；
3. 加载特征标准化器；
4. 加载 ONNX 模型；
5. 逐帧读取视频；
6. 提取人脸关键点；
7. 计算疲劳相关特征；
8. 更新时间窗口缓存；
9. 构造当前窗口特征向量；
10. 输入模型进行状态分类；
11. 显示结果并触发告警；
12. 循环执行直到程序退出。

### 4.2 在线检测流程
在线检测阶段以实时性为核心，处理重点包括：
- 关键点稳定提取；
- 特征快速计算；
- 推理结果平滑；
- 状态变化及时显示；
- 疲劳状态触发报警。

### 4.3 离线训练流程
离线训练阶段流程如下：

1. 采集正常与疲劳状态视频；
2. 提取每一帧的人脸关键点；
3. 计算单帧疲劳特征；
4. 按滑动窗口构造训练样本；
5. 生成标签并划分数据集；
6. 训练轻量 MLP；
7. 保存模型参数和标准化器；
8. 导出 ONNX 模型；
9. 使用 ONNX Runtime 验证推理结果。

## 6. 模型设计

### 6.1 分类任务定义

优先采用二分类任务：

* `0`：正常
* `1`：疲劳风险

这样实现更稳，训练难度更低，也更适合课程作业展示。

### 6.2 MLP 结构设计

建议采用一个小型全连接网络，结构如下：

```python
Linear(feature_dim, 32)
ReLU()
Linear(32, 16)
ReLU()
Linear(16, num_classes)
```

说明：

* `feature_dim` 为输入特征维度；
* 隐藏层较小，有利于边缘端运行；
* 使用 ReLU 激活函数；
* 输出层对应二分类结果。

### 6.3 输入特征标准化

使用训练集拟合标准化参数，对输入特征做统一尺度变换，以提高模型训练稳定性和推理一致性。

### 6.4 训练参数建议

* 损失函数：CrossEntropyLoss
* 优化器：Adam
* 学习率：0.001
* Batch Size：32 或 64
* Epoch：30～80

## 7. 数据集与样本构建

### 7.1 数据采集方式

采用自采集方式录制视频，重点采集以下状态：

* 正常注视；
* 闭眼；
* 打哈欠；
* 低头或偏头；
* 多种疲劳行为混合。

### 7.2 样本构造方式

假设滑动窗口长度为 `N=15` 帧：

* 每 15 帧构成一个样本；
* 滑动步长可设为 3 或 5；
* 每个样本由一个时间窗口内的统计特征表示；
* 样本标签由该时间窗口整体状态确定。

### 7.3 样本格式示例

```text
sample_id, ear_l, ear_r, ear_avg, mar, pitch, yaw, roll,
ear_mean, ear_min, mar_mean, mar_max,
low_ear_count, high_mar_count, close_eye_ratio, yawn_ratio, label
```

## 8. 关键实现细节

### 8.1 疲劳特征计算

系统核心特征包括：

* EAR：用于表示眼睛开合程度；
* MAR：用于表示嘴巴张开程度；
* 头姿角：用于辅助判断低头或偏头状态；
* 时间窗口统计值：用于增强连续疲劳行为识别能力。

### 8.2 状态平滑

在线检测过程中，单帧波动较大，因此建议增加状态平滑机制，例如：

* 连续若干次预测为疲劳才触发报警；
* 最近若干帧结果多数投票决定当前状态；
* 对置信度设置最小阈值。

### 8.3 异常情况处理

系统还需考虑以下异常情况：

* 未检测到人脸；
* 多人同时出现；
* 光照不足；
* 用户距离摄像头过远。

可采用以下处理策略：

* 未检测到人脸时暂停判定；
* 默认只处理主人脸；
* 在界面上提示“未检测到人脸”。

## 9. ONNX 导出与部署

### 9.1 导出流程

1. 训练 PyTorch MLP 模型；
2. 构造示例输入张量；
3. 使用 `torch.onnx.export()` 导出 ONNX 模型；
4. 使用 ONNX Runtime 加载模型进行验证。

### 9.2 推理流程

```text
窗口特征向量 -> 标准化 -> float32 转换 -> ONNX Runtime 推理 -> 输出类别 -> 状态显示
```

### 9.3 部署意义

将模型导出为 ONNX 并使用 ONNX Runtime 推理，能够体现：

* 模型训练与部署分离；
* 部署流程更规范；
* 系统更贴近边缘智能应用；
* 更便于后续迁移到其他平台。

## 10. 实验设计

### 10.1 实验目标

实验主要验证以下三个问题：

1. 该系统能否有效识别疲劳状态；
2. 轻量 MLP 是否优于简单人工规则；
3. ONNX Runtime 是否有助于提升本地推理效率。

### 10.2 实验一：分类效果评估

评价指标包括：

* Accuracy
* Precision
* Recall
* F1
* 混淆矩阵

实验内容：

* 统计测试集分类结果；
* 分析正常与疲劳状态的识别情况；
* 观察是否存在明显误报或漏报。

### 10.3 实验二：人工规则与轻量模型对比

人工规则可采用：

* EAR 低于阈值且持续若干帧；
* MAR 高于阈值且持续若干帧；
* 组合条件触发疲劳风险。

轻量模型则基于多维疲劳特征进行学习和判断。

对比内容：

* 准确率；
* 稳定性；
* 对个体差异的适应能力。

### 10.4 实验三：部署前后推理性能对比

对比对象：

* 原始 PyTorch 推理；
* ONNX Runtime 推理。

对比指标：

* 平均单帧推理时间；
* 平均 FPS；
* 模型文件大小；
* 内存占用情况（可选）。

### 10.5 实验四：特征组合分析

可分析不同特征组合对识别效果的影响，例如：

* 仅使用 EAR；
* 使用 EAR + MAR；
* 使用 EAR + MAR + 头姿；
* 加入时间窗口统计特征。

## 13. 风险分析与解决思路

### 13.1 风险点

* 数据量较少，模型泛化能力有限；
* 关键点检测受环境影响较大；
* 不同个体之间面部特征差异明显；
* 在线检测存在瞬时误报。

### 13.2 解决思路

* 扩充采集场景，提高数据多样性；
* 使用状态平滑减少误报；
* 对特征进行标准化处理；
* 在实验中分析个体差异带来的影响。
